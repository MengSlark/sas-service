# SAS 路径改写与执行

使用此 Skill 处理“先改路径，再执行 SAS”的请求。

本 Skill 是一个单一能力，但内部必须按阶段工作：

1. 分析 SAS 代码中的输入路径、输出路径、动态路径构造点。
2. 生成路径改写计划。
3. 仅在高置信前提下改写路径，不改业务逻辑。
4. 对改写结果做自检。
5. 仅当自检通过时，调用 `/execute` 执行并下载全部产物。

## 何时使用

当用户的意图属于以下类型时，使用此 Skill：

- 执行 `.sas` 文件
- 运行 SAS 程序
- 先按项目目录规范改写路径，再执行 SAS
- 将 SAS 适配到测试目录、开发目录或统一输出目录后执行
- 执行 SAS 并下载全部产物

当用户只是想了解 SAS 逻辑、解释代码、讨论算法，且没有要求改写路径或执行时，不使用此 Skill。

## 核心原则

- 默认先改路径，再执行。
- 路径改写由模型完成，不依赖专门的路径改写脚本。
- 改写仅限路径相关内容，不得修改业务逻辑。
- 若任一关键路径无法高置信判断，必须中止执行。
- 宁可拒绝执行，也不要在低置信前提下冒险改写并提交。

## 工作流

### 第一阶段：识别

先读取原始 `.sas` 文件内容，只做分析，不立即改代码。

需要识别的内容包括：

- 所有输入 `.xpt` 路径
- 所有输出路径
- 所有可能影响路径的宏变量、字符串拼接、条件分支
- 项目名称候选
- 无法静态判断的动态路径构造点

此阶段的目标是得到“路径清单”和“风险清单”。

### 第二阶段：项目名称推断

项目名称按以下优先级确定：

1. 用户当前请求中明确给出的项目名称
2. 当前会话中最近一次提到的项目名称

如果存在多个候选且无法唯一确定，必须停止执行并提示用户确认。


## 时间戳规则

在执行前，必须生成一个实际时间戳，格式为 `YYYYmmdd_HHMMSS`。

要求如下：

- 代码内所有输出路径使用同一个时间戳
- `/execute` 的 `output_dir` 使用同一个时间戳
- 不允许代码内输出目录和提交给 `/execute` 的输出目录不一致

## 目标目录规范

以下不是示例，而是执行时必须使用的真实路径模板。

输入前缀固定为：

`D:\SAS_Data\TestD\development\<项目名称>\input`

其中只有 `<项目名称>` 可以替换。

输出前缀固定为：

`D:\SAS_Data\TestD\development\<项目名称>\output-YYYYmmdd_HHMMSS`

其中只有 `<项目名称>` 和时间戳 `YYYYmmdd_HHMMSS` 可以替换。

除上述替换外，不得改成其他根目录或其他目录结构。

输入 `.xpt` 分类规则：

- 文件名以 `ad` 开头，归入 `input\adam\`
- 其他 `.xpt` 归入 `input\sdtm\`

示例：

- `adsl.xpt` -> `input\adam\adsl.xpt`
- `advs.xpt` -> `input\adam\advs.xpt`
- `ae.xpt` -> `input\sdtm\ae.xpt`

输出路径统一规则：

- ADaM 类数据文件（如 `.xpt`、`.json`、`.sas7bdat`）写到 `输出前缀\data\adam\`
- TLF 相关数据文件或中间数据写到 `输出前缀\data\tlf\`
- `rtf` 报表写到 `输出前缀\report\`

## 路径改写要求

模型在改写时必须遵守以下规则：

- 只改路径，不改业务逻辑
- 不改 `libname` 名称
- 不改宏变量名称
- 不改数据集名称
- 不改业务判断、数据处理、统计逻辑

在改写任何路径之前，必须先判断该路径或宏变量的语义角色，至少区分以下三类：

- 项目根目录
- 输入根目录
- 输出根目录

如果某个宏变量表示项目根目录，例如变量名、注释或上下文语义明确指向 `projectdir`、`rootdir`、`projdir`、项目文件夹路径或类似含义，则该变量默认视为“项目根目录变量”。

项目根目录变量的值必须保持为：

`D:\SAS_Data\TestD\development\<项目名称>\`

对于这类变量：

- 不得改写为 `D:\SAS_Data\TestD\development\<项目名称>\input`
- 不得改写为 `D:\SAS_Data\TestD\development\<项目名称>\output-YYYYmmdd_HHMMSS`
- 不得直接替换成任何具体输入目录或具体输出目录

如果输入路径和输出路径都依赖同一个项目根目录变量，必须保留该变量为项目根目录，只能改写其引用处，不得改写该变量本身的根目录语义。

路径改写应优先修改“引用处的后缀结构”，而不是优先修改“根目录变量定义”。

例如：

- `&projectdir.input\adam` 属于输入路径引用，应改为 `D:\SAS_Data\TestD\development\<项目名称>\input\adam`
- `&projectdir.output\data\tlf` 属于输出路径引用,应该为 `D:\SAS_Data\TestD\development\<项目名称>\output-YYYYmmdd_HHMMSS\data\tlf`

对于上述情形，应根据目标目录规范改写引用处的完整落点；但 `projectdir` 这类项目根目录变量本身不得被改写为输出目录变量。

以下路径形式都应被纳入分析范围：

- `~/` 或 `~\`
- 相对路径
- 仅文件名
- 绝对路径
- UNC 路径
- 宏变量参与拼接的路径
- 通过函数或表达式构造的路径

但并不是所有情况都必须强行改写。

- 原始位置
- 无法高置信改写的原因
- 建议用户补充的信息

## 结构化改写计划

在真正改写代码前，模型应先在内部形成一个结构化改写计划。计划至少应包含：

- 原始输入路径或文件名
- 目标输入路径
- 原始输出路径
- 目标输出路径
- 项目名称
- 时间戳
- 每个改写项的理由
- 每个改写项的置信度
- 未解决风险项

只有在不存在关键风险项时，才允许进入实际改写。

## 改写后检查

执行前至少检查以下条件：

- 代码中不应再出现 `~/` 或 `~\`
- 代码中出现的 `.xpt` 输入路径都应落在：
  - `D:\SAS_Data\TestD\development\<项目名称>\input\adam\`
  - 或 `D:\SAS_Data\TestD\development\<项目名称>\input\sdtm\`
- 代码中所有输出文件路径都应落在同一个 `输出前缀` 下
- ADaM 类数据文件应落在 `输出前缀\data\adam\`
- TLF 相关数据文件或中间数据应落在 `输出前缀\data\tlf\`
- `rtf` 报表应落在 `输出前缀\report\`
- `/execute` 使用的 `output_dir` 必须与代码内输出前缀一致
- 改写前后业务逻辑应保持不变

## 执行方式

只有在改写和检查都通过后，才调用现有执行脚本：

```bash
python scripts/execute_and_download.py --code-file <rewritten.sas> --output-dir D:\SAS_Data\TestD\development\<项目名称>\output-YYYYmmdd_HHMMSS
```

可选参数：

- `--base-url`，默认：`http://115.190.133.229:8080`
- `--output-dir`，执行输出目录
- `--timeout`，默认：`600`

## 结果判定

执行完成后，应综合日志和产物判断结果：

- 日志中没有明显 `error`，且存在除日志以外的输出产物，且产物正常，可判定为执行成功
- 日志中有明显 `error`，且只有日志没有其他产物，或产物数据为空，应判定为执行失败
- 如果服务端返回成功，只代表sas执行完成，但日志和产物存在冲突，应明确提示存在结果不一致风险

## 输出约定

脚本输出 JSON，例如：

```json
{
  "success": true,
  "request_id": "abc123def456",
  "saved_files": ["C:\\path\\file1.rtf", "C:\\path\\file2.json"],
  "artifacts": [...]
}
```

## 对用户的反馈要求

执行此 Skill 时，对用户的反馈应尽量简洁，但必须包含以下关键信息：

- 本次采用的项目名称
- 本次采用的时间戳
- 是否已完成路径改写
- 若未执行，阻断原因是什么
- 若已执行，执行结果和下载到本地的产物概况

## 说明

- 文件下载到当前工作目录
- 若文件重名，会自动添加数字后缀
- 项目输出目录必须是 `D:\SAS_Data\TestD\development\<项目名称>\output-YYYYmmdd_HHMMSS`
- 所有下载文件(artifacts)都应作为本次任务的产出物
